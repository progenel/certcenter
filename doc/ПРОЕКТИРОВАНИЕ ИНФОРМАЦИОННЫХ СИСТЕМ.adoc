*ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННЫХ СИСТЕМ.*

*Цель проекта:*

Закрепить навыки анализа требований, структурирования функционала,
построения UML-диаграмм и разработки архитектуры информационной системы
на примере выбранной темы. Студенты должны пройти полный цикл
проектирования: от формулирования требований до разработки прототипа и
презентации итогов работы.

*1. Темы для проектов:*

. Разработка геймификационной платформы для проверки и обучения знаниям
в области кибербезопасности
. Разработка системы контроля версий документов и учета изменений в
организации
. Разработка системы анализа логов и обнаружения аномалий для
предприятия
. Разработка прототипа средства мониторинга безопасности SCADA-систем
. Использование криптографических методов для защиты данных на малых и
средних предприятиях
. Разработка интеллектуальной системы мониторинга сетевого трафика и
обнаружения атак с использованием машинного обучения.

*2. Общие требования к проекту*

Каждая группа должна:

*Провести полный цикл проектирования:** анализ предметной области;

* выявление пользователей и ролей системы;
* определение функциональных и нефункциональных требований;
* формирование архитектуры системы;
* построение UML-диаграмм;
* разработка прототипа интерфейса (Figma, Miro, Drawio, etc.);
* подготовка технической документации;
* защита проекта.

**Работа выполняется группой из 3–5 человек,**с распределением ролей:
аналитик, архитектор, дизайнер UI, инженер данных, документатор.

*3. Содержание проекта*

*3.1 Введение*

Группа должна подготовить короткий аналитический блок:* описание
проблемы;

* актуальность;
* краткий обзор существующих решений;
* обоснование выбора технологии/подхода.

*3.2 Анализ требований*

*3.2.1. Определение ролей пользователей*

_Не менее 3 ролей. Например:_* Администратор

* Пользователь
* Модератор
* Инженер безопасности
* Аналитик логов

и т.д. (в зависимости от темы)

*3.2.2. Функциональные требования (FR)*

_Минимум 10–15 функций._

_Пример формата:_

FR-01: Пользователь должен иметь возможность проходить тестовые задания.

FR-02: Система должна фиксировать историю действий пользователя.

*3.2.3. Нефункциональные требования (NFR)*

_Минимум 8–10._

_Например:_* NFR-01 Производительность: обработка запросов до 200 мс

* NFR-02 Безопасность: двухфакторная аутентификация
* NFR-03 Масштабируемость: до 1000 одновременных пользователей

*3.3. Разработка архитектуры*

_Студенты должны:_

Описать логическую архитектуру* уровни (UI, бизнес-логика, БД,
интеграции);

* ключевые компоненты;
* технология стека.

Создать архитектурную диаграмму

_(например: C4 model – Context / Container / Component)_

*3.4. UML-диаграммы (обязательно)*

_Каждая группа выполняет не менее 6 типов UML-диаграмм:_

*Use-case диаграмма (обязательная)*

Покрыть весь функционал.

Диаграмма классов

Диаграмма последовательности (Sequence)

Диаграмма деятельности (Activity)

Диаграмма состояний (State Machine)

Диаграмма компонентов или развертывания (Component / Deployment)

_Все диаграммы должны быть оформлены в инструменте:_* _Draw.io_

* _StarUML_
* _Visual Paradigm_
* _PlantUML_
* _Miro_

*(на выбор)*

*3.5. Проект базы данных*

Группа разрабатывает:

*ER-диаграмму*

с сущностями, связями и cardinality.

*Описание таблиц:** название

* атрибуты
* ключи
* связи

_При необходимости — нормализация (1NF–3NF)._

*3.6. Прототип интерфейса (UI/UX)*

_Каждая группа готовит:_

Интерфейс минимум 5 экранов:* Главная страница

* Страница авторизации
* Основной рабочий экран
* Экран настроек / админ-панели
* Экран просмотра данных / отчетности

*_Инструменты: Figma, Miro, Balsamiq._*

*3.7. Разработка прототипа (MVP)*

_Рекомендуется:_

Реализовать небольшой рабочий прототип:* форма входа

* мини-модуль (пример: загрузка логов / прохождение теста / регистрация
документа)
* демонстрационный экран аналитики или мониторинга
